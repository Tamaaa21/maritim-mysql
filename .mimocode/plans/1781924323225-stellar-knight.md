### 2. Duplicate Helper Functions
- `getRole()`, `getUsername()`, `getUserId()` duplicated in:
  - `app/api/admin/users/route.ts`
  - `app/api/admin/users/[id]/route.ts`
- These already exist in `services/admin.service.ts` ‚Äî should reuse

### 3. Inconsistent Error Handling
- Some routes: `serverError(error)` from `lib/response.ts`
- Others: raw `NextResponse.json({ success: false, message: "..." })`
- `catch (error: any)` instead of `catch (error: unknown)`

### 4. Architecture
- No shared validation middleware ‚Äî each route validates independently
- Mixed patterns: some routes use services, others directly access DB
- Display route PATCH has complex reorder logic that could be extracted

## Implementation Plan

### Step 1: Fix `any` types in API routes
- Add proper types for DB query results using existing types from `lib/types.ts`
- Create local interfaces where needed (e.g., BMKG API response types)
- Replace `catch (error: any)` with `catch (error: unknown)` + type narrowing

### Step 2: Remove duplicate helpers
- Delete local `getRole()`, `getUsername()`, `getUserId()` from `users/route.ts` and `users/[id]/route.ts`
- Import from `services/admin.service.ts` instead

### Step 3: Standardize error handling
- All routes use `serverError()` from `lib/response.ts`
- Replace `catch (error: any)` with proper error handling

### Step 4: Extract display reorder logic
- Move reorder logic from `display/route.ts` PATCH into a service function
